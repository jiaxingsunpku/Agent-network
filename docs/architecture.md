# 架构设计：智能体网络平台（ANP）

本文是「新系统怎么搭」的权威说明。命名细节见 [naming.md](naming.md)，消息契约见 protocol.md，状态定义见 world-status.md，网关接口见 gateway-api.md（后三者随 P0 第二批补全）。

## 1. 定位

智能体网络平台是基于 Kafka 的**分布式黑板系统**，为某一类协同任务定义智能体之间的协作约定、信息交换标准与时空统一性。

- 智能体「只发布自己产生的信息，只订阅自己所需的信息」；平台不干预智能体内部业务逻辑，只提供协作基础设施。
- 平台**不是大而统的中枢大脑**。早期「把所有领域智能体所需信息全量汇总到一个统一世界模型」的设想已被否定：信息冗余高、算力带宽浪费、不利于分布式扩展。
- 一个**协同任务域 = 一个 World Model**（如「交通管控 World Model」）。一个 World Model 在逻辑上对应完整任务域，在物理上由一组关联 Topic + 系统级智能体 + 存储 + 管理规则组成。平台可承载多个 World Model，靠命名体系隔离（见 naming.md）。

## 2. 三类智能体

| 类别 | 职责 | 边界 |
|---|---|---|
| **感知智能体** | 采集物理世界信息并初步处理，向平台上报世界状态 | 自带模型与算力，独立维护；只需遵守发布协议；不做复杂推理与决策 |
| **系统级智能体**（平台内置） | 维护全局世界状态：对离散感知数据做窗口聚合、清洗、对齐、计算，生成语义化 World Status 写入状态层 Topic | **所有共识性、基础性指标（排队长度、路段流量等）都由它算**，不为此另起独立任务智能体；用时间窗口处理网络延迟、对齐物理/语义时间、过滤过期与异常上报 |
| **任务/执行智能体** | 订阅 World Status 完成业务决策、推理与执行 | 业务逻辑自实现；平台只供状态输入、管理生命周期、（后续）给奖励反馈。细分为侧重分析的「任务智能体」与侧重改变物理世界的「执行智能体」 |

**系统级智能体 / World Status 状态层是本次重构的核心**，也是老仓库缺失的部分——老仓库只把感知数据原样搬运，没有这一层语义聚合。

## 3. 分层数据流（黑板 + 闭环）

```
        ┌─────────────────────────────────────────────┐
  决策层 │  任务/执行智能体   （订阅 World Status）        │
        └───────▲───────────────────────────┬─────────┘
                │ World Status               │ 命令下发
                │（状态层 topic）             │（控制层 topic）
        ┌───────┴───────────────────────────│─────────┐
  平台   │  系统级智能体（内置）               │          │
  核心   │  窗口聚合 / 清洗 / 对齐 / 算指标     │          │
        │  + World Status 当前态（供网关读）   │          │
        └───────▲───────────────────────────│─────────┘
                │ 原始观测 + 状态镜像          │
                │（感知层 topic）             ▼
        ┌───────┴─────────────────────────────────────┐
  感知层 │  感知智能体（边缘）  执行效果经此回流          │
        └─────────────────────────────────────────────┘

  网关（FastAPI，读模型）：订阅状态层 → snapshot/projection；命令入口 → 控制层
  前端：只读网关 HTTP，不直连 Kafka
```

闭环：感知上行 → 系统级智能体算出 World Status → 决策层订阅并决策 → 命令下行到边缘执行 → 执行效果被感知层采集再次上行。

## 4. 模块职责（backend/anp/）

职责单一，根治老仓库「一个 server.py 又算又投影又收命令」的混乱：

| 模块 | 职责 | 不做 |
|---|---|---|
| `contracts/` | **唯一契约源**：envelope、topic 常量、三类智能体角色、命令/ack 的 Pydantic 模型与校验 | 不含业务逻辑 |
| `system_agent/` | 消费感知层 → 时间窗口聚合/清洗/对齐/过滤 → 产出 World Status 写状态层 → 维护「每实体最新状态」 | 不直接服务前端 HTTP |
| `gateway/` | 纯读模型：订阅状态层构建 snapshot/projection；命令入口校验+白名单+发布到控制层 | **不做窗口聚合/世界状态计算** |
| `registry/` | 智能体注册、心跳、在线/降级/离线状态、白名单（谁能发/订哪些 topic） | 不做业务决策 |
| `adapters/` | 感知/执行智能体接入适配（沿用老仓库 bridge 思想，但拆干净、契约统一）；详见 [adapters.md](adapters.md)。已落地 `signalvision/`（仅感知侧） | 不在此散搓 envelope、不算 World Status |

`backend/agents/` 放可独立运行的示例智能体；v1 是虚拟交通智能体（既当感知体上报观测，又当执行体接命令）。

## 5. 系统级智能体 v1 具体做什么（不空谈）

域：交通（traffic）。

1. 消费感知层观测（各方向车辆数、滞留车辆数、速度等）。
2. 按 **N 秒滚动时间窗口**、以路口为 key 聚合。
3. 计算共识指标：
   - 排队长度 ← 滞留车辆数；
   - 路段流量（veh/h）← 窗口内车辆数换算；
   - 平均延误、平均速度。
4. 过滤：事件时间早于窗口水位线的迟到消息、不合 schema 的上报、被标为异常的数据源，直接丢弃。
5. 产出 World Status（每路口一条语义状态）写状态层 topic，并在内存维护「每路口最新 World Status」供网关读取。

窗口大小、字段、状态 schema 的精确定义放 world-status.md。

## 6. 网关读模型 + 复用 UI 的关键原则

**网关对外 HTTP 契约保持与老仓库前端兼容**，背后换成全新分层实现。这样前端可以照搬，只改 API base 指向新网关 + 对齐少量字段，视觉完全不动。

老仓库前端已消费的接口（新网关对齐）：

```
GET  /api/agent-network/snapshot
GET  /api/agent-network/projection?kind={node|edge|resource}&id={id}
POST /api/agent-network/commands          # 必带 target_agent_id，禁止 broadcast/agent_ids
GET  /api/agent-network/timeseries/*      # 冷路径，本期返回「未启用」占位
```

- snapshot 的节点来自 World Status（路口）+ registry（智能体）；边来自路网连接；资源来自来源（检测器）与去向（控制器/库）。
- 命令入口校验后发布到控制层 topic；协作端本地 Safety Guard 仍是权威安全闭环；网关只协调与审计。
- 网关不可用或无有效数据时，前端沿用老仓库的 mock 回落（`source mock`）。

精确接口契约放 gateway-api.md。

## 7. 范围与阶段

本期只重构后端 + 恢复 UI（范围见 [AGENTS.md](../AGENTS.md) §1）。阶段：

- **P0** 治理脚手架 + 高层设计文档（本批）→ 详细契约文档（第二批）。
- **P1** `contracts/` + `deploy/` 单节点 Kafka + topic 清单。
- **P2** 系统级智能体 + 虚拟感知体，产出真实 World Status。
- **P3** 网关读模型 + 命令入口，契约对齐前端。
- **P4** 迁移前端，指向新网关，端到端跑通（非 mock）。
- **P5** 真实 SignalVision adapter：**感知接入已落地**（`adapters/signalvision/`，见 [adapters.md](adapters.md)）；命令控制闭环（信号控制）/ 视频流 / 路口预测三件会议任务仍属后续。

## 8. 从老仓库借鉴 vs 丢弃

- **借鉴**：UI 全套；Kafka envelope 与 topic 思想；Safety Guard 留在协作端；虚拟智能体闭环；docker-compose 单节点 Kafka；三类智能体划分；冷路径区分。
- **丢弃**：巨石 `ui/server.py`（又算又投影又收命令）；「大而统世界模型」框架；各处手搓 envelope；`agent-network` / `world-model` 双轨命名；内存态易丢的注册逻辑。
